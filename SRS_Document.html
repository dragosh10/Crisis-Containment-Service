1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) document provides a comprehensive description of the Crisis Containment Service (CRI) system. The document is intended for stakeholders including developers, project managers, testers, and end-users to understand the system's functionality, constraints, and requirements.

1.2 Scope

The Crisis Containment Service (CRI) is a web-based disaster management and alert system designed to:

  • Provide real-time disaster monitoring and visualization
  • Enable authorities to manage and report disasters
  • Allow citizens to set up location-based alerts
  • Facilitate emergency shelter management
  • Generate CAP (Common Alerting Protocol) alerts for affected users
  • Provide educational resources for disaster preparedness

1.3 Definitions, Acronyms, and Abbreviations

Term Definition
CRI Crisis Containment Service - The name of the disaster management system
CAP Common Alerting Protocol - A standard format for emergency alerts
SRS Software Requirements Specification
API Application Programming Interface
CRUD Create, Read, Update, Delete operations
WebSocket Protocol for real-time bidirectional communication

2. Overall Description

2.1 Product Perspective

The CRI system is a standalone web application that integrates with external APIs and services:

  • External APIs: USGS Earthquake API, UK Environment Agency Flood API, Nasa Fires API
  • Database: MySQL database for storing user data, disasters, and alerts
  • Real-time Communication: WebSocket server for live alerts
  • Alert Generation: CAP XML alert generation system

System Architecture

Frontend: HTML5, CSS3, JavaScript
Backend: Node.js
Database: MySQL
Maps: Leaflet.js
Real-time: WebSocket
Security: bcrypt
Alerts: XML Builder

2.2 Product Functions

The major functions of the CRI system include:

User Management

  • User registration and authentication
  • Role-based access control (Authority vs Civilian)
  • Session management

Disaster Management

  • Real-time disaster data visualization
  • Disaster reporting and management by authorities
  • Multi-source disaster data integration
  • Disaster filtering and search capabilities

Alert System

  • Location-based alert subscription
  • Automatic CAP alert generation
  • Real-time alert delivery via WebSocket
  • Alert history and management

Shelter Management

  • Emergency shelter location management
  • Shelter capacity and type tracking
  • Shelter-disaster association

2.3 User Characteristics

Authority Users

  • Role: Emergency services, government officials
  • Technical Expertise: Moderate to high
  • Responsibilities: Disaster reporting, shelter management, alert coordination
  • Access Level: Full system access including CRUD operations

Civilian Users

  • Role: General public
  • Technical Expertise: Basic to moderate
  • Responsibilities: Setting alert preferences, viewing disaster information
  • Access Level: Read-only access with personal alert management

2.4 Constraints

Technical Constraints

  • Web browser compatibility requirements
  • Internet connectivity dependency
  • External API rate limiting
  • Real-time performance requirements

Regulatory Constraints

  • Data privacy regulations (GDPR compliance)
  • Emergency alert standards (CAP protocol)
  • Accessibility requirements

Operational Constraints

  • 24/7 system availability requirement
  • Multi-language support considerations
  • Mobile device compatibility

2.5 Assumptions and Dependencies

Assumptions

  • Users have access to modern web browsers
  • Reliable internet connectivity is available
  • External APIs remain accessible and stable

Dependencies

  • External disaster data APIs (USGS, UK Environment Agency, Nasa)

3. Specific Requirements

3.1 Functional Requirements

3.1.1 User Authentication and Authorization

REQ-001: User Registration
Description: The system shall allow users to register with email and password
Priority: High
Input: Email address, password
Output: User account creation confirmation
Processing: Validate email format, hash password, store in database
REQ-002: User Authentication
Description: The system shall authenticate users using email and password
Priority: High
Input: Email address, password
Output: Authentication success/failure, session token
Processing: Verify credentials, create session cookie
REQ-003: Role-Based Access Control
Description: The system shall differentiate between authority and civilian users
Priority: High
Input: User role (determined by email domain @cri.com for authorities)
Output: Appropriate interface and permissions
Processing: Check user role and redirect to appropriate dashboard

3.1.2 Disaster Data Management

REQ-004: Real-time Disaster Data Display
Description: The system shall display real-time disaster data on an interactive map
Priority: High
Input: External API data (earthquakes, floods, fires)
Output: Visual markers on map with disaster information
Processing: Fetch API data, render map markers
REQ-005: Disaster Reporting (Authority Only)
Description: Authority users shall be able to report new disasters
Priority: High
Input: Disaster coordinates, type, description, severity
Output: Disaster record creation, alert generation
Processing: Validate input, store in database, trigger alert system
REQ-006: Disaster Filtering
Description: Users shall be able to filter disasters by type and location
Priority: Medium
Input: Filter criteria (disaster type, location)
Output: Filtered disaster display
Processing: Apply filters to dataset, update map display

3.1.3 Alert System

REQ-007: Location-Based Alert Subscription
Description: Civilian users shall be able to set up to 3 location-based alert pins
Priority: High
Input: GPS coordinates, location name
Output: Alert pin creation confirmation
Processing: Validate coordinates, store in clientPins table
REQ-008: Automatic Alert Generation
Description: The system shall automatically generate CAP alerts for users within 30km of disasters
Priority: High
Input: New disaster data, user pin locations
Output: CAP XML alert files, real-time notifications
Processing: Calculate distances, generate CAP XML, send WebSocket alerts
REQ-009: Real-time Alert Delivery
Description: The system shall deliver alerts in real-time using WebSocket
Priority: High
Input: Alert data, user connection
Output: Real-time alert notification
Processing: Maintain WebSocket connections, broadcast alerts

3.1.4 Shelter Management

REQ-010: Shelter Location Management
Description: Authority users shall be able to manage emergency shelter locations
Priority: Medium
Input: Shelter coordinates, type, capacity, description
Output: Shelter record creation/update
Processing: Validate input, store in shelters table
REQ-011: Shelter Visualization
Description: All users shall be able to view shelter locations on the map
Priority: Medium
Input: Shelter data from database
Output: Shelter markers on map
Processing: Fetch shelter data, render map markers

3.1.5 Educational Resources

REQ-012: Disaster Preparedness Tips
Description: The system shall provide educational content about disaster preparedness
Priority: Low
Input: User request for tips
Output: Educational content display
Processing: Display static educational content with navigation

3.2 Non-Functional Requirements

3.2.1 Performance Requirements

NFR-001: Response Time
Description: The system shall respond to user requests within 10 seconds
Priority: High
Measurement: Average response time for API calls and page loads
NFR-002: Real-time Alert Delivery
Description: Alerts shall be delivered within 5 seconds of disaster reporting
Priority: High
Measurement: Time from disaster creation to alert delivery

3.2.2 Security Requirements

NFR-004: Password Security
Description: User passwords shall be hashed using bcrypt with salt rounds ≥ 10
Priority: High
Measurement: Code review of password hashing implementation
NFR-005: Session Management
Description: User sessions shall expire after 24 hours of inactivity
Priority: High
Measurement: Session timeout testing
NFR-006: Input Validation
Description: All user inputs shall be validated and sanitized
Priority: Medium
Measurement: Security testing for XSS and SQL injection

3.2.3 Usability Requirements

NFR-007: Responsive Design
Description: The system shall be usable on devices with screen widths ≥ 320px
Priority: Medium
Measurement: Testing on various device sizes
NFR-008: Browser Compatibility
Description: The system shall work on Chrome, Firefox and Edge (latest 2 versions)
Priority: Medium
Measurement: Cross-browser testing

3.2.4 Reliability Requirements

NFR-009: System Availability
Description: The system shall have 99.5% uptime
Priority: High
Measurement: Uptime monitoring over 30-day periods
NFR-010: Error Handling
Description: The system shall gracefully handle API failures and display appropriate error messages
Priority: Medium
Measurement: Error scenario testing

4. System Models

4.1 Use Case Diagrams

Use Case 1: User Registration and Login

Actors: Civilian User, Authority User

Preconditions: User has access to the system

Main Flow:

  1. User navigates to registration page
  2. User enters email and password
  3. System validates input and creates account
  4. User logs in with credentials
  5. System authenticates and redirects to appropriate dashboard

Postconditions: User is authenticated and has access to system features

Use Case 2: Disaster Reporting (Authority)

Actors: Authority User

Preconditions: Authority user is logged in

Main Flow:

  1. Authority user clicks on map to select location
  2. User fills disaster report form
  3. System validates and saves disaster data
  4. System generates alerts for affected users
  5. System displays confirmation message

Postconditions: Disaster is recorded and alerts are sent

Use Case 3: Setting Alert Pins (Civilian)

Actors: Civilian User

Preconditions: Civilian user is logged in

Main Flow:

  1. User selects "Place Pin" option
  2. User clicks on map to set coordinates
  3. User enters pin name
  4. System validates and saves pin location
  5. System displays pin on map

Postconditions: User will receive alerts for disasters near this location

4.2 Data Model

Database Schema

Table Description Key Fields
users User account information id, email, password, is_authority
calamities Disaster records id, lat, lng, type, description, gravity
clientPins User alert locations id, id_client, pin1_lat, pin1_lng, pin1_name
shelters Emergency shelter locations id, lat, lng, type_shelter, permanent
user_alerts Generated alerts for users id, user_id, event, instruction, lat, lon

4.3 System Architecture

Three-Tier Architecture

  • Presentation Layer: HTML/CSS/JavaScript frontend with Leaflet.js maps
  • Application Layer: Node.js server with Express.js framework
  • Data Layer: MySQL database with connection pooling

External Integrations

  • USGS Earthquake API: Real-time earthquake data
  • Nasa Fires Api: Real-time fires data
  • UK Environment Agency API: Flood monitoring data
  • WebSocket Server: Real-time alert delivery
  • CAP Alert Generator: Standard emergency alert format

5. Appendices

5.1 Glossary

Term Definition
Disaster Pin A location marker set by users (authorities) to send alerts for disasters
Personal Pin User-defined location markers (up to 3 per civilian user) for receiving personalized disaster alerts based on proximity to important locations such as home, work, or family
Shelter Pin Location markers indicating emergency shelter facilities managed by authorities, displaying shelter type, capacity, and availability for disaster response
Authority Dashboard Administrative interface for managing disasters and shelters
Civilian Dashboard User interface for viewing disasters and managing alert preferences
CAP Alert Common Alerting Protocol formatted emergency notification
Real-time Alert Immediate notification delivered via WebSocket connection

5.2 Technology Stack Details

Frontend Technologies

  • HTML5: Semantic markup and structure
  • CSS3: Styling and responsive design
  • JavaScript (ES6+): Client-side logic and interactivity

  • Font Awesome: Icon library

Backend Technologies

  • Node.js: Server-side JavaScript runtime
